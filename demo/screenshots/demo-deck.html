The Problem: Brittle Automation

Every enterprise has repetitive browser workflows. ServiceNow tickets, Jira tasks, insurance claims, HR onboarding. Today they're either done manually or automated with fragile scripts.

What breaks scripts:

  • UI redesign changes CSS selectors
  • Button text changes ("Save" to "Apply")
  • New confirmation modal appears
  • Table columns get reordered
  • Form fields move to different page

The cost:

  • $50-400B/year per vertical in manual services
  • RPA implementations fail 30-50% of the time
  • Every UI update = weeks of script maintenance
  • Companies hire humans instead of fixing scripts

The gap: Scripted automation = consulting gig (rebuild every time). Self-healing automation = scalable product.

The Solution: Intent-Based Automation

Instead of telling the computer HOW to interact (selectors), tell it WHAT to achieve (goals). The system figures out the HOW at runtime.

Traditional (Scripted)

page.click('#btn-resolve-main')

page.fill('#comment-textarea', text)

page.click('button.submit-btn')

Selector changes = script dies

TierZero (Intent-Based)

goal: "Click the Resolve button"

goal: "Fill the comment field"

goal: "Submit the form"

UI changes = system adapts

Key insight: The workflow describes the GOAL ("click Resolve"), not the SELECTOR ("#btn-resolve"). When the UI changes, the IntentEngine finds the element by understanding the page, not memorizing paths.

→

▶️ ReplayAdaptive execution
  • Record: CDP captures every click, keystroke, and navigation with page state snapshots
  • Annotate: LLM generates semantic descriptions ("Clicked the Search button in the nav bar") and detects variable vs constant values
  • Generate: Converts annotated recording into intent-based workflow steps - goals, not selectors
  • Package: Outputs a hot-loadable TierZero skill with skill.json manifest and executable entry point
  • Replay: IntentEngine executes each intent adaptively with a 5-strategy fallback chain + recovery

No coding required. A business analyst demonstrates the task. TierZero generates the automation. If the UI changes, the automation adapts.

Now: The UI Gets Redesigned

The product team ships a redesign. Traditional automation breaks immediately. Let's see what changed:

  • Button text changed: "Resolve" → "✓ Done", "Add Comment" → "💬 Comment", "Save Changes" → "💾 Save"
  • Button position changed: Moved from bottom-right to top and bottom-left
  • CSS classes changed: .btn → .action-btn, .card → .panel
  • Table columns reordered: Priority moved to first column, Status to last
  • Labels uppercased: "Username" → "USERNAME", "Add Comment" → "ADD COMMENT"
  • Color scheme changed: Blue → Purple/Indigo
  • New confirmation modal: v2 shows "Are you sure?" when resolving

With traditional scripting: Every selector fails. Every text match fails. Script is dead. Engineer spends days rewriting.

With TierZero: Workflow says "Click Resolve button." IntentEngine finds "✓ Done" via aria-label. Zero changes needed.

How the IntentEngine Adapts

Five resolution strategies tried in order. If one fails, the next kicks in:

  • 1. Cached (~50ms): Try the last-known-good selector. If the UI hasn't changed, this is instant.
  • 2. Aria (~200ms): Find by accessibility role + label. "Resolve" button found even when visible text is "✓ Done" because aria-label is preserved.
  • 3. LLM - Accessibility Tree (~500ms): Feed the page's accessibility tree to GPT-4o-mini. It understands context: "this looks like a submit/resolve action button."
  • 4. LLM - Vision (~1s): Screenshot the page, ask GPT-4o vision to locate the element. Works even with completely custom UIs.
  • 5. Coordinates (~1.5s): Ask the LLM for pixel coordinates and click directly. Last resort fallback.

Recovery strategies:

  • Dialog Dismissal: Detects and closes unexpected modals/alerts automatically
  • LLM Recovery: Analyzes page state, suggests navigation or wait actions
  • Page State Assertions: Verifies expected state between steps, retries if needed
